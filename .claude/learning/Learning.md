### 2026-06-27: 服务重启后 spawn 会话应自动重连

- **场景**：cc-remote 服务端在 `restoreState()` 恢复持久化状态时，spawn 模式的 Claude Code 会话对应的子进程已在服务重启时被杀死。
- **错误**：`restoreState()` 仅恢复了内存中的 session 对象，未检查 spawn 会话的进程是否存活，导致用户看到的会话显示为"已连接"但实际无进程响应。
- **根因**：`restoreState()` 缺少对 spawn 会话的生命周期感知——持久化状态仅保存元数据，不保存进程句柄。服务重启后，必须在恢复阶段主动重新拉起子进程。
- **教训**：在 `restoreState()` 中，对每个 spawn 模式的会话应自动调用 `spawnConnectSession()` 传入 `--resume` 参数重新拉起 Claude Code 进程。**凡涉及子进程的持久化恢复，必须显式处理进程重建逻辑**，不能假设进程仍然存活。

### 2026-06-27: takeoverSession() 必须以 JSONL 文件为权威事实源刷新消息缓存

- **场景**：用户在远程端通过 spawn 会话与 Claude Code 对话后，回到主机端直接对话（不经过 cc-remote），然后再从远程端重连该会话。重连时消息列表未包含主机端直接对话期间产生的新消息。
- **错误**：`takeoverSession()` 在接管已有会话时，直接复用了内存中 `session.historyMessages` 缓存的旧消息列表，未重新读取磁盘上的 JSONL 文件。
- **根因**：设计上假定 cc-remote 是会话的唯一写入者，忽略了用户可能绕过 cc-remote 直接操作 Claude Code（主机终端直接对话）的场景。内存缓存与磁盘文件产生了不一致，而代码信任了内存缓存。
- **教训**：`takeoverSession()` 接管会话时必须重新读取 JSONL 文件刷新 `historyMessages`，以 JSONL 文件为**权威事实源**，内存缓存仅作为性能优化，不可作为可信状态。**凡是存在外部写入可能的数据源，读取路径必须在关键节点（如会话接管、恢复）重新从磁盘加载，而非信任内存缓存**。
